## 각 문서 핵심 요약

### 01 전체 파일 구조
프로젝트는 **재료 준비 → 카메라 공격 → 드론 영향 분석 → 드론 학습 → 성능 평가** 5단계로 흐른다.
17개 Python 파일이 각 단계를 담당하며, 앞 단계의 결과 파일(JSON/PTH)을 다음 단계가 읽어가는 파이프라인 구조다.
최종 결과는 `results/final_summary.png` 1장에 6개 그래프로 압축된다.

### 02 CE 임계값
CE(오염 경험) 차단은 **이진(10px 초과 전량 폐기)** 과 **연속 가중치(`max(0.05, 1.0 - shift/50)`)** 두 방식으로 실험했다.
분모 50은 실험에서 관측된 shift 최대 의미 범위, 최솟값 0.05는 완전 차단을 막는 안전장치다.
연속 가중치가 이진 차단보다 유효 경험을 4배 이상 남겨(6,393개 vs 27,873개) 도착률이 31%→65%(V4) / 49%→78%(V5)로 향상됐다.

### 03 보상 함수
도착 +1.0 / 충돌 -1.0은 대칭 설계로 두 결과의 무게를 같게 만드는 기준점이다.
시간 페널티 -0.01은 200걸음 다 써도 합계가 -0.99여서 "느리게라도 도착하는 것"이 여전히 이익이 되는 크기다.
거리 보상 ±0.01은 시간 페널티와 같은 크기로 맞춰서, 잘 가면 합계 0점(상쇄), 잘못 가면 -0.02(이중 패널티)가 되도록 설계됐다.

### 04 FGSM vs PGD 차이
FGSM과 PGD의 코드 차이는 딱 한 줄 — `for _ in range(steps):` 루프의 유무다.
`step_size = epsilon/4`는 루프 1번째에 경계에 붙어버리지 않고 10번에 걸쳐 경계 안을 탐색하게 만드는 값이다.
실험 결과 PGD가 FGSM보다 2.4~3.6배 강하며, epsilon이 클수록 탐색 공간이 넓어져 배율이 더 커진다.

### 05 V3 실패 원인 분석
V3는 8개 장애물 + 고정 경로(1.0,1.0)→(13.5,16.5)로 항상 같은 좁은 틈만 연습해 ep~400에서 47%를 찍고 붕괴했다.
리플레이 메모리(10,000개 한도)가 실패 경험으로 채워지면서 성공 경험이 밀려나 결국 0%가 됐다(Catastrophic Forgetting).
`reset()`에서 `list(START_V3)` / `list(GOAL_V3)` → `_random_free_pos()` 두 줄 교체만으로 74%→65%(V4-B)가 됐다.

### 06 V6 오염 페널티
V5-B의 "오염 경험 저장 차단"에 한 가지를 더 추가했다 — 오염이 심한 상황에서 행동하면 보상에서 추가 페널티를 깎는다.
RISK(shift 10~20px, 장애물 3km 이내 접근 중): -0.01 / CRITICAL(shift >20px, 무조건): -0.03.
1차 시도에서 RISK=-0.02가 너무 자주 발동(36%/스텝)해 학습 붕괴가 일어났고, 조건 강화 + 크기 축소 후 V5-B 78% → V6 79%(+1%p)가 됐다.

### 07 V7 DDM 버퍼
V6 위에 ImprovedDDMBuffer를 연결해 경험을 RE/DE/SE/CE 4종류로 분류하고 종류별 서브큐에 따로 저장한다.
꺼낼 때는 RE:DE:SE = 4:3:3 비율로 섞어 중요 경험이 더 많이 학습에 쓰이도록 한다.
CE(shift>10px)가 전체 스텝의 80.8%를 차지해 유효 경험이 V5-B의 1/7 수준으로 줄었고, 최종 도착률이 55%(V6 79% 대비 -24%p)로 낮았다.
이는 CE 임계값(10px)이 PGD ε=0.03 평균 shift(13.8px)보다 낮아서 정상 경험까지 오염으로 오분류된 결과다.

# 01 전체 파일 구조

# FRDDM-DQN 프로젝트 전체 파일 구조 설명

> **프로젝트 한 줄 요약:**
> 드론이 AI 카메라에 의존해 날아다니는데, 누군가 카메라를 속이면 드론이 어떻게 되는지 실험하고, 해결책을 만들어 검증하는 프로젝트.

---

## 전체 흐름

```
사진 만들기
  → AI 눈 학습
    → 공격 실험 (FGSM / PGD)
      → 영향 분석 (드론 항법 오차)
        → 해결책 설계 (CE 필터)
          → 드론 학습 (v2 → v3 → v4 → v5)
            → 성능 평가 (Confusion Matrix, 비교표)
              → 최종 정리 (final_summary.png)
```

---

## 1단계: 재료 준비

---

### `src/generate_images.py`

**이 파일이 하는 일:** 군용 차량이 있는 가짜 항공 사진 5장을 컴퓨터로 그려서 만든다.

**입력:** 없음 (아무것도 받지 않고 처음부터 그림)

**출력:** `data/images/` 폴더에 차량 그림 5장 (vehicle_01.png ~ vehicle_05.png)

**중요한 변수:**
- `IMG_W`, `IMG_H` = 1000, 500 → 사진 크기(가로 1000픽셀, 세로 500픽셀)를 결정
- `VEHICLE_COLORS` → 차량을 어떤 초록색으로 칠할지 결정 (군용 위장색 모음)

---

### `src/train_frcnn.py`

**이 파일이 하는 일:** AI가 사진에서 차량을 찾아 네모 박스(바운딩 박스)를 치는 방법을 학습시킨다.

**입력:** `data/images/` 폴더의 차량 사진, `data/gt_annotations.json` (정답 박스 위치)

**출력:** `results/frcnn_finetuned.pth` (학습된 AI 눈), `results/bbox_clean.json` (공격 없을 때 박스 결과)

**중요한 변수:**
- `NUM_EPOCHS = 3` → AI를 몇 번 반복해서 공부시킬지 (3번)
- `SCORE_THRESH = 0.3` → "이 정도 이상 확신할 때만 박스를 그려" (30% 이상)

---

## 2단계: 카메라 공격 실험

---

### `src/attack.py`

**이 파일이 하는 일:** 학습된 AI 카메라에 FGSM과 PGD 두 가지 방법으로 이미지를 살짝 바꿔서 박스가 얼마나 틀려지는지 측정한다.

**입력:** `results/frcnn_finetuned.pth` (학습된 AI 눈), 차량 사진 5장

**출력:** `results/attack_results.json` (FGSM 결과), `results/attack_results_pgd.json` (PGD 결과), 그래프 PNG 2개

**중요한 변수:**
- `EPSILONS = [0, 0.01, 0.03, 0.05, 0.07, 0.1]` → 공격 세기 목록 (0=공격없음, 0.1=가장 강한 공격)
- `PGD_STEPS = 10` → PGD 공격은 10번 반복해서 조금씩 이미지를 망가뜨림

---

### `src/detection_failure.py`

**이 파일이 하는 일:** 공격받은 사진에서 AI가 차량을 못 찾거나(Miss), 없는 차량을 봤다고 하는(Ghost) 비율을 계산한다.

**입력:** `results/attack_results.json` (FGSM), `results/attack_results_pgd.json` (PGD), `data/gt_annotations.json` (정답)

**출력:** `results/detection_failure.json` (실패율 숫자), `results/detection_failure.png` (그래프)

**중요한 변수:**
- `IOU_THRESH = 0.5` → 박스가 정답과 절반 이상 겹쳐야 "맞다"고 판정 (50% 기준)
- `"miss_cnt"`, `"ghost_cnt"` → 각 사진·각 공격 세기별로 실제로 몇 번 실패했는지 기록

---

## 3단계: 오차가 드론에 미치는 영향 분석

---

### `src/analyze.py`

**이 파일이 하는 일:** 박스가 틀려진 픽셀 오차를 드론의 실제 거리·각도 오차로 환산해서 "공격이 드론 항법에 얼마나 위험한지" 수치로 보여준다.

**입력:** `results/attack_results.json` (FGSM 결과)

**출력:** `results/state_vector_error.png` (드론 상태 오차 그래프)

**중요한 변수:**
- `TAU = 0.05` → 픽셀을 실제 거리로 바꾸는 스케일 계수 (논문 수식에서 가져온 값)
- `EPSILONS` → 공격 세기별로 오차가 얼마나 커지는지 비교

---

### `src/improved_ddm.py`

**이 파일이 하는 일:** "공격받은 경험은 학습에서 걸러내자"는 새로운 판단 규칙(개선된 DDM)을 코드로 설계한다.

**입력:** 없음 (다른 파일들이 이 파일을 도서관처럼 가져다 씀)

**출력:** 없음 (코드 부품 모음 — `DQNAgent`, `DDMState` 등 클래스 제공)

**중요한 변수:**
- `CE_THRESH_PX = 10.0` → 박스가 10픽셀 이상 밀렸으면 "오염된 경험"으로 판정
- `RISK_DIST` → 장애물에서 이 거리 안에 들어오면 위험 상태로 간주

---

### `src/summarize.py`

**이 파일이 하는 일:** FGSM 공격 결과를 4개 그래프로 한 장에 요약해 보여준다 (초기 버전 요약 그림).

**입력:** `results/attack_results.json`

**출력:** `results/final_summary.png` (초기 요약 그래프 — 나중에 덮어씌워짐)

**중요한 변수:**
- `TAU = 0.05` → analyze.py와 같은 스케일 계수
- 서브플롯 4개 → bbox 오차, 거리 오차, 각도 오차, 신뢰도 변화

---

## 4단계: 드론 학습 실험

---

### `src/dqn_comparison.py`

**이 파일이 하는 일:** 공격받는 환경에서 "기존 방식(모두 저장)"과 "개선 방식(오염 경험 차단)"을 처음으로 비교 학습한다.

**입력:** 없음 (환경 직접 생성)

**출력:** `results/dqn_comparison.png`, `results/dqn_results.json`, `DQNAgent`·`pgd_attack` 등 부품 제공

**중요한 변수:**
- `MAX_STEPS = 200` → 드론이 한 번 시도에서 최대 200걸음만 걸을 수 있음
- `CE_THRESH = 10.0` → 박스가 10픽셀 이상 틀렸으면 그 경험은 배우지 않음

---

### `src/visualize_flight.py`

**이 파일이 하는 일:** 기존 DDM과 개선 DDM 드론이 실제로 어떤 경로로 날아다녔는지 10개 시나리오를 그림으로 비교한다.

**입력:** 없음 (내부에서 드론을 직접 학습·실행)

**출력:** `results/flight_paths.png` (10행 × 2열 비행 경로 그림)

**중요한 변수:**
- `N_TEST = 10` → 10가지 다른 출발지·목적지로 테스트
- `OBS_RADIUS = 1.5` → 장애물 반경 1.5km 안에 들어오면 충돌 판정

### `src/dqn_comparison_v3.py`

**이 파일이 하는 일:** 장애물을 5개→8개로 늘리고 "연속 가중치 CE"(오염 정도에 따라 조금씩 배우는 방식)를 처음 실험한다.

**입력:** 없음

**출력:** `results/dqn_comparison_v3.png`, `results/dqn_results_v3.json`

**중요한 변수:**

- 장애물 고정 배치 `(1.0,1.0)→(13.5,16.5)` → 항상 같은 좁은 길을 통과해야 함

### `src/env_test.py`

- `OBSTACLES_8` → 8개 장애물 위치 목록 (다른 모든 v4·v5 실험이 이 목록을 그대로 씀)

## 5단계: 성능 평가

---

### `src/ce_evaluation.py`

**이 파일이 하는 일:** "CE 필터가 진짜 나쁜 경험을 잘 걸러내는가?"를 숫자로 평가한다 — 임계값 전략 비교표와 순환논리 없는 Confusion Matrix를 만든다.

**입력:** `results/dqn_results_v4.json`, `results/dqn_results_v5.json`, `results/env_test_results.json`, `results/detection_failure.json`, `results/attack_results_pgd.json`

# 02 CE 임계값

# CE 임계값(threshold) 설명

> **핵심 개념을 한 줄로:**
> 드론 카메라가 공격을 받아 박스가 X픽셀 이상 흔들렸을 때,
> "이 경험은 오염됐으니 배우지 말자"고 결정하는 기준선이 바로 CE 임계값이다.

---

## 질문 1: 임계값을 10px, 20px, 30px로 바꾸면 코드 안에서 정확히 무엇이 달라지는가?

### 먼저, shift가 뭔지부터

드론은 매 걸음마다 카메라로 앞을 찍고, AI가 "저기 장애물이 있어!"라는 박스를 그린다.
PGD 공격이 들어오면 이 박스가 원래 위치에서 **옆으로 밀린다**.
그 밀린 거리(픽셀)가 바로 `shift`다.

```
정상일 때:   박스 중심 (316px, 257px)
공격 받을 때: 박스 중심 (98px, 480px)  ← 엄청 멀리 이동
                              ↑
                          shift = sqrt((316-98)² + (257-480)²) ≈ 290px
```

우리 실험(ε=0.03)에서 shift 평균은 **약 13.8px**, 표준편차 4px이다.

---

### 방법 A: 이진 차단 (dqn_comparison_v4.py, V4-A)

**임계값이 있는 코드 (66번째 줄):**

```python
# dqn_comparison_v4.py
CE_BINARY_THRESH  = 10.0   # ← 이 숫자 하나를 바꾸면 된다
```

**이 숫자를 쓰는 함수 (75~76번째 줄):**

```python
def _deposit_ratio_binary(shift: float) -> float:
    return 0.0 if shift > CE_BINARY_THRESH else 1.0
```

**학습 루프 안에서 실제로 일어나는 일 (133~146번째 줄):**

```python
ratio = deposit_fn(shift)          # 위 함수 호출 → 0.0 또는 1.0

should_deposit = (ratio > 0.0)     # 비이진 방식에선 random, 이진에선 단순 비교

if should_deposit:
    agent.push(...)    # 리플레이 메모리에 저장 (나중에 이걸로 학습함)
    valid_exp += 1
else:
    ce_blocked += 1    # 그냥 버림
```

**임계값을 바꾸면 이 한 줄이 달라진다:**

| 임계값 설정 | `shift = 13px`일 때 `ratio`값 | 결과 |
|------------|-------------------------------|------|
| `CE_BINARY_THRESH = 10.0` | `0.0` (10보다 크니까 버림) | 이 경험 폐기 |
| `CE_BINARY_THRESH = 20.0` | `1.0` (20보다 작으니까 저장) | 이 경험 저장 |
| `CE_BINARY_THRESH = 30.0` | `1.0` (30보다 작으니까 저장) | 이 경험 저장 |

**각 임계값에서 예상 폐기율 (PGD 평균 shift=13.8px, 표준편차=4px 기준):**

| 임계값 | 의미 | 예상 폐기율 |
|--------|------|------------|
| 10px | "아주 조금만 움직여도 버린다" | ~82% 폐기 (V4-A 실제 측정: **82.8%**) |
| 20px | "꽤 많이 움직인 것만 버린다" | ~50% 폐기 (예상치) |
| 30px | "심하게 움직인 것만 버린다" | ~10% 폐기 (예상치) |

> 10px으로 설정한 이유: attack.py 실험에서 ε=0.01부터 이미 평균 12.9px 오차가 발생함을 확인했기 때문이다. 10px은 "공격이 조금이라도 시작된 것"을 잡아내는 조기 감지선이다.

---

### 방법 B: 개선된 DDM의 2단계 임계값 (improved_ddm.py)

이진 차단과 달리, `improved_ddm.py`는 임계값이 **두 개**다.
"좀 이상해" (RISK)와 "완전히 망가짐" (CRITICAL)을 따로 구분한다.

**두 임계값 정의 (95~98번째 줄):**

```python
# improved_ddm.py
_SHIFT_THRESHOLD_RISK     = 10.0   # "좀 이상해" 경계선
_SHIFT_THRESHOLD_CRITICAL = 20.0   # "완전히 망가짐" 경계선
```

**이 숫자들을 쓰는 판정 함수 (137~141번째 줄):**

```python
if shift > _SHIFT_THRESHOLD_CRITICAL:   # 20px 초과?
    level = CorruptionLevel.CRITICAL     # → "심각한 오염"
elif shift > _SHIFT_THRESHOLD_RISK and level is CorruptionLevel.STABLE:
    level = CorruptionLevel.RISK         # → "경미한 오염"
# 아무 조건도 해당 안 되면: STABLE (정상)
```

**임계값을 바꾸면 shift=13px인 경험이 어떻게 분류되는지:**

| `RISK` 경계 | `CRITICAL` 경계 | `shift=13px` 판정 |
|-------------|----------------|-------------------|
| 10px | 20px (현재 설정) | **RISK** (10<13<20이니까) |
| 20px | 30px로 바꾸면 | **STABLE** (13<20이니까) |
| 5px  | 15px로 바꾸면 | **CRITICAL** (13<15이지만 >5이므로 RISK→CRITICAL) |

**판정 결과에 따라 달라지는 행동:**

```python
# DDM_DEPOSIT_RATIO (160~167번째 줄)
DDM_DEPOSIT_RATIO = {
    ExperienceType.RE: 1.00,   # 목표 도달·충돌 결과: 전량 저장
    ExperienceType.DE: 0.80,   # 장애물 근접: 80% 저장
    ExperienceType.SE: 0.20,   # 안전 이동: 20% 저장
    ExperienceType.CE: 0.00,   # 오염 경험: 완전 차단
}
```

RISK나 CRITICAL 둘 다 CE로 분류되어 **저장율 0.00** → 완전 폐기.

## 질문 2: 연속 가중치 수식 `max(0.05, 1.0 - shift/50.0)` 해설

### 수식이 하는 일을 그림으로

```
저장 확률
1.0 |●
    |  ＼
    |    ＼
0.5 |      ＼
    |        ＼
    |          ＼
0.05|____________●━━━━━━━━
    +─────────────────────→ shift (px)
    0   10   25   50   80
                  ↑
              여기서부터 최솟값 고정
```

**실제 코드 (dqn_comparison_v4.py, 67·79~80번째 줄):**

```python
CE_SHIFT_MAX = 50.0   # 분모

def _deposit_ratio_continuous(shift: float) -> float:
    return max(0.05, 1.0 - shift / CE_SHIFT_MAX)
```

**shift 값별 저장 확률 계산 예시:**

| shift | 계산 과정 | 저장 확률 | 의미 |
|-------|-----------|-----------|------|
| 0px  | max(0.05, 1.0 - 0/50) = 1.00 | **100%** | 공격 없음 → 무조건 저장 |
| 5px  | max(0.05, 1.0 - 5/50) = 0.90 | **90%** | 아주 경미한 흔들림 |
| 13.8px | max(0.05, 1.0 - 13.8/50) = 0.724 | **72%** | 실험 평균 공격 세기 |
| 25px | max(0.05, 1.0 - 25/50) = 0.50 | **50%** | 꽤 강한 공격 |
| 50px | max(0.05, 1.0 - 50/50) = max(0.05, 0) = 0.05 | **5%** | 최대 공격 수준 |
| 80px | max(0.05, 1.0 - 80/50) = max(0.05, -0.6) = 0.05 | **5%** | 그 이상도 최솟값 유지 |

---

### 왜 50으로 나누는가?

**50이라는 숫자의 근거: "실험에서 관측된 shift의 최대 의미 범위"**

`attack_results_pgd.json` 실험 결과를 보면:
- ε=0.01: shift 평균 ~31px
- ε=0.03: shift 평균 ~37px
- ε=0.07: shift 평균 ~43px
- 측정된 최대 shift: **~61px** (vehicle_04, ε=0.03)

50px를 분모로 쓰면:
- `shift = 50px`일 때 비율이 정확히 0이 되어 → `max(0.05, 0) = 0.05`
- 즉 "50px 이상이면 이제 최솟값만 남긴다"는 기준점이 된다
- 50px은 "공격이 매우 심하지만, 그래도 완전히 버리지는 않을 상한선"으로 설계된 것

**만약 50 대신 다른 숫자를 썼다면:**

| 분모 값 | shift=13.8px에서의 저장 확률 | 결과 |
|---------|---------------------------|------|
| 20 (너무 작게) | max(0.05, 1 - 13.8/20) = **0.31** (31%) | 평균 공격에서도 70% 폐기 → V4-A 이진과 비슷해짐, 유효 경험 부족 |
| **50 (현재 설정)** | max(0.05, 1 - 13.8/50) = **0.724** (72%) | 평균 공격에서 72% 저장 → 유효 경험 충분 |
| 100 (너무 크게) | max(0.05, 1 - 13.8/100) = **0.862** (86%) | 심한 공격도 86% 저장 → 오염 경험이 너무 많이 섞임 |

**V4/V5 실험 결과로 확인:**
- V4-B (50 사용): 유효 경험 **27,873개**, 최종 도착률 **65%**
- V4-A (이진 10px): 유효 경험 **6,393개**, 최종 도착률 **31%**

50이라는 분모 덕분에 "공격을 어느 정도 걸러내면서도 학습에 충분한 경험을 남기는" 균형이 맞았다.

---

### 왜 최솟값을 0.05(5%)로 정했는가?

**0.05라는 숫자의 근거: "완전히 0으로 만들지 않겠다"는 설계 철학**

```python
return max(0.05, 1.0 - shift / CE_SHIFT_MAX)
#          ↑
#       이 값 덕분에 shift가 아무리 커도
#       저장 확률이 0%가 되지 않는다
```

**만약 0.05 대신 다른 최솟값을 썼다면:**

| 최솟값 | 의미 | 예상 결과 |
|--------|------|-----------|
| **0.00** (최솟값 없음) | shift > 50px이면 저장 확률 0% → 이진 차단과 같아짐 | V4-A처럼 유효 경험이 급감 |
| **0.05** (현재 설정) | 가장 심한 공격도 5%는 저장 | 26,530개 유효 경험 확보 (V5-B) |
| **0.20** (너무 크게) | 심한 공격도 최소 20% 저장 | 오염 경험이 너무 많이 유입 |

**0.05의 두 가지 역할:**

1. **학습 붕괴 방지**: 완전히 배움을 차단하면, 드론이 "장애물 근처에서 카메라가 흔들릴 때 어떻게 해야 하는가"를 아예 못 배움. 5%라도 남겨두면 그 상황에 대한 약한 학습 신호가 유지됨.

2. **폐기율 균형**: 0.05가 없으면 shift > 50px 구간에서 **수식이 음수**가 된다 (`1.0 - 80/50 = -0.6`). `max(0.05, 음수) = 0.05`가 이 음수를 막아주는 안전장치이기도 하다.

**V5 실험으로 검증:**
- V5-B (0.05 사용): 폐기율 **27.4%**, 도착률 **78%** (최고 성능)
- V5-A (이진 차단, 최솟값 개념 없음): 폐기율 **82.8%**, 도착률 **49%**

"조금이라도 남긴다(0.05)"는 선택이 도착률을 **+29%p** 높이는 데 기여했다.

## 두 질문 합쳐서 보는 전체 그림

```
매 걸음마다:

  카메라 박스 흔들림 측정
         ↓
       shift = ?px
         ↓
  ┌─────────────────────────────────────────┐
  │  이진 차단 (V4-A, V5-A)                 │
  │  if shift > 10px → 버림 (저장확률 0%)   │
  │  else            → 저장 (저장확률 100%) │
  └─────────────────────────────────────────┘
            vs
  ┌─────────────────────────────────────────┐

         ↓                    ↓
  유효 경험 6,393개     유효 경험 26,530개
  최종 도착률 49%       최종 도착률 78%
  (V5-A 기준)          (V5-B 기준)
```

핵심: **이진 차단은 단호하지만 너무 많이 버린다. 연속 가중치는 오염 정도에 비례해 조금씩 걸러내서 학습 재료를 더 많이 남긴다.**

# 03 보상 함수

# 보상 함수 설계 설명

> **핵심 개념을 한 줄로:**
> 보상 함수는 드론에게 주는 "점수 규칙"이다.
> 어떤 행동을 했을 때 몇 점을 주는지 설계하는 것이
> 드론이 "어떤 방식으로 날기를 선호하는가"를 결정한다.

---

## 보상 구조 전체 그림

```
매 걸음마다 드론이 받는 점수:

  ┌─────────────────────────────────────────────────────┐
  │ 종료 상황 (에피소드 끝)                              │
  │   목적지 도달  → reward = +1.0   ← 이번 걸음 끝    │
  │   충돌·이탈    → reward = -1.0   ← 이번 걸음 끝    │
  ├─────────────────────────────────────────────────────┤
  │ 비종료 상황 (계속 날아야 함, V5에서 추가)            │
  │   기본값       →          0.0                       │
  │ + 시간 페널티  →         -0.01  (매 걸음 무조건)    │
  │ + 거리 보상    →   +0.01 or -0.01  (방향에 따라)   │
  └─────────────────────────────────────────────────────┘
```

**실제 코드 (env_test.py 184~189번째 줄 + dqn_comparison_v5.py 127~136번째 줄):**

```python
# env_test.py — 기본 보상 (V4까지 사용)
if goal_reached:
    reward = 1.0          # 도착!
elif collision or out_of_bounds:
    reward = -1.0         # 충돌 or 경계 이탈
else:
    reward = 0.0          # 평범한 걸음

# dqn_comparison_v5.py — V5에서 추가
if not (info["goal_reached"] or info["collision"] or info["out_of_bounds"]):
    reward += TIME_PENALTY          # -0.01 (비종료 걸음마다)
    curr_dist = math.dist(env.pos, env.goal)
    if curr_dist < prev_dist:
        reward += DIST_APPROACH     # +0.01 (목적지에 가까워짐)
    elif curr_dist > prev_dist:
        reward += DIST_RECEDE       # -0.01 (목적지에서 멀어짐)
```

## 숫자 1: 도착 보상 +1.0

### 왜 +1.0인가?

이 값은 전체 보상 체계의 **기준점(anchor)** 이다.
DQN은 이 +1.0을 "가장 좋은 일이 일어났다"는 신호로 학습한다.

```
Q(상태, 행동) → 미래에 받을 점수 예측값
```

- +1.0이 크면: 드론이 도착을 "강하게 원하게" 된다
- +1.0이 작으면: 도착이 별로 좋은 일이 아닌 것처럼 느껴진다

**1.0 대신 다른 값을 쓰면:**
- `+10.0`으로 키우면: 이론적으로는 같지만, 나머지 보상들(±0.01)이 상대적으로 너무 작아져서 학습 신호로서 의미가 없어진다. "도착만 중요하고, 경로는 아무래도 좋다"는 드론이 된다.
- `+0.1`으로 줄이면: 거리 보상(±0.01)이 상대적으로 커진다. 드론이 목적지 근처에서 왔다갔다하며 조금씩 점수를 긁어모으는 이상한 행동을 학습할 수 있다.

---

## 숫자 2: 충돌·이탈 페널티 -1.0

### 왜 -1.0인가? 왜 +1.0과 정확히 같은 크기인가?

```
도착 보상: +1.0
충돌 패널: -1.0
────────────────
차이:       2.0
```

이 대칭 구조가 중요하다.
드론 입장에서 "목적지 도달"과 "충돌"은 **정반대의 결과**이기 때문에,
똑같은 절댓값으로 보상과 패널티를 주면 두 상황의 무게가 같아진다.

**만약 비대칭으로 바꾼다면:**

| 설정 | 예상 문제 |
|------|-----------|
| 도착 +1.0, 충돌 -0.1 | 드론이 충돌을 별로 무서워하지 않음. "어차피 충돌해도 -0.1밖에 안 깎이잖아" |
| 도착 +1.0, 충돌 -5.0 | 드론이 충돌이 너무 무서워 아예 움직이지 않거나, 목적지 근처에서도 멈추는 지나치게 보수적인 행동 |

---

## 숫자 3: 시간 페널티 -0.01 (V5에서 추가)

### 왜 -0.01인가?

**먼저, 왜 시간 페널티가 아예 필요했는가?**

V4 실험 결과에서 드론이 장애물 주위를 빙빙 돌다 200걸음 제한에 걸리는 문제가 있었다.
V4의 비종료 걸음 보상은 0.0 — 아무 점수도 없으니, 드론은 "돌아도 그만, 전진해도 그만"이었다.

시간 페널티를 추가하면:
- 매 걸음이 손해 → 드론이 빨리 끝내려는 동기 생김
- 결국 목적지로 직진하는 전략이 더 유리해짐

**왜 하필 -0.01인가? (크기 비율의 핵심)**

`-0.01`이라는 크기를 고르기 위해 세 가지 비율을 계산해봐야 한다.

```
최대 걸음 수 × 시간 페널티 = 200 × (-0.01) = -2.0
도착 보상                                    = +1.0
```

200걸음을 다 써도 도착 보상이 여전히 시간 페널티 합계보다 **작다**.
이 비율이 바로 핵심이다.

**"200걸음을 써서라도 도착하는 것"이 여전히 이익인가?**

시나리오를 수치로 비교해보자.

```
경우 A: 50걸음 만에 목적지 도달 (직진)
  비종료 걸음 49개 × (-0.01) = -0.49
  + 도착 보상                = +1.00
  ─────────────────────────────────
  합계                       = +0.51  ← 이익

경우 B: 200걸음 다 쓰고 목적지 도달 (빙빙 돌다 간신히)
  비종료 걸음 199개 × (-0.01) = -1.99
  + 도착 보상                  = +1.00
  ─────────────────────────────────
  합계                         = -0.99  ← 손해! 충돌(-1.0)과 거의 같음

경우 C: 50걸음에 충돌
  비종료 걸음 49개 × (-0.01) = -0.49
  + 충돌 패널티              = -1.00
  ─────────────────────────────────
  합계                       = -1.49  ← 가장 나쁨
```

이 비율의 결론: **"빠른 도착 > 느린 도착 > 충돌"** 순으로 학습된다.

**만약 -0.01이 아니라 다른 값을 썼다면:**

| 시간 페널티 | 200걸음 도착 시 합계 | 문제 |
|------------|---------------------|------|
| `-0.001` (너무 작음) | 199×(-0.001)+1.0 = +0.80 | 빙빙 돌아도 손해가 거의 없음 → V4 문제 그대로 |
| **`-0.01` (현재)** | **199×(-0.01)+1.0 = -0.99** | 느린 도착이 손해 → 빠른 경로 선호 유도 |
| `-0.05` (너무 큼) | 199×(-0.05)+1.0 = -8.95 | 느린 도착이 충돌(-1.0)보다 훨씬 나쁨 → 드론이 빠르게 이탈하는 게 더 유리해짐 → V5-A의 이탈 현상과 유사 |

---

## 숫자 4: 거리 보상/페널티 +0.01 / -0.01 (V5에서 추가)

### 왜 ±0.01인가? 왜 시간 페널티와 같은 크기인가?

**이 값을 시간 페널티(-0.01)와 맞춘 이유가 있다.**

두 값이 같을 때 걸음별 합계가 어떻게 되는지 계산해보자.

```
목적지를 향해 한 걸음 갔을 때:
  시간 페널티  = -0.01
  거리 보상    = +0.01
  ─────────────────────
  합계         =  0.00  ← 제자리

목적지 반대 방향으로 한 걸음 갔을 때:
  시간 페널티  = -0.01
  거리 패널티  = -0.01
  ─────────────────────
  합계         = -0.02  ← 두 배 손해
```

이 설계의 핵심:
- **잘 가면 0점** — 패널티를 상쇄할 뿐, 보상은 없다
- **잘못 가면 -0.02** — 시간 낭비 + 방향 실수 이중 패널티

"잘 가도 보상이 없다"는 게 이상해 보일 수 있다.
하지만 이렇게 설계하면 "목적지 쪽으로 가는 것"이 기본값이 되고,
"최종 도착 +1.0"이 진짜 보상이 된다.

**만약 거리 보상을 더 크게 했다면 (`+0.1 / -0.1`):**

```
목적지 방향으로 한 걸음:  -0.01 + 0.1 = +0.09 (꽤 큰 보상)
반대 방향으로 한 걸음:   -0.01 - 0.1 = -0.11
```

이렇게 되면 드론이 장애물을 무시하고 목적지 쪽으로 돌진하다 충돌할 수 있다.
"장애물을 돌아가면 거리가 일시적으로 멀어지는데", 그 구간에서 -0.1씩 받으면
"돌아가지 말고 그냥 뚫고 가자"는 전략을 배울 수 있다.

**만약 거리 보상을 더 작게 했다면 (`+0.001 / -0.001`):**

- 시간 페널티(-0.01)가 거리 보상보다 10배 크다
- 드론이 "어느 방향이든 빨리 끝내자"라는 신호를 받음
- 장애물 밀집 구간에서 목적지와 반대 방향으로 빠르게 이탈하는 게 더 유리해질 수 있음

---

## 비율이 왜 중요한가 — 한 번에 정리

4개 보상 값의 크기 관계를 위계로 정리하면:

```
보상 크기 위계:

도착  +1.00  ████████████████████  ← 가장 강한 신호
충돌  -1.00  ████████████████████  (대칭)

────────────────────────────────────── (100배 차이)

거리  ±0.01  █                     ← 방향 유도 신호
시간  -0.01  █                     (같은 크기)
```

**왜 100배 차이를 두는가?**

DQN은 미래 보상을 GAMMA(0.95)로 할인해 합산한다.
```
총 예상 보상 = 지금 보상 + 0.95×(다음 보상) + 0.95²×(다다음 보상) + ...
```

GAMMA=0.95에서 할인 효과:

| 몇 걸음 뒤 | 할인 계수 | 50걸음 뒤의 +1.0은? |
|-----------|-----------|---------------------|
| 1걸음 뒤 | 1.00 | — |
| 10걸음 뒤 | 0.95⁹ ≈ 0.63 | — |
| 50걸음 뒤 | 0.95⁴⁹ ≈ 0.08 | 0.08점 |
| 100걸음 뒤 | 0.95⁹⁹ ≈ 0.006 | 0.006점 |

100걸음 뒤의 도착 +1.0은 지금 기준으로 0.006점밖에 안 된다.
이 때 **거리 보상 ±0.01이 매 걸음마다 즉각 지급**되므로, 드론이 멀리 있는 도착 보상을 기다리지 않고도 **지금 어느 방향으로 갈지** 배울 수 있다.

이것을 **"밀집 보상(dense reward)"** 이라고 부른다.
8장애물 환경처럼 목적지에 도달하기 어려운 환경에서는
드물게 주어지는 도착 보상만으로는 학습이 너무 느려진다.
작은 거리 보상이 매 걸음 신호를 주기 때문에 수렴이 빨라진다.

## V4 vs V5 실험 결과로 확인

| 실험 | 보상 추가 | V-A 도착률 | V-B 도착률 |
|------|-----------|-----------|-----------|
| V4 | 없음 (비종료 = 0점) | 31% | 65% |
| V5 | 시간 -0.01 + 거리 ±0.01 | 49% | **78%** |

V5-B가 V4-B보다 **+13%p** 향상된 것이 보상 설계의 효과다.
V5-A도 31%→49%로 향상됐지만, 이진 CE로 유효 경험이 부족해
"보상 신호는 좋아졌지만 배울 경험이 없는" 상태가 됐다.

## 최종 요약 — 보상 설계의 3가지 원칙

1. **종료 보상 = 진짜 목표 신호 (±1.0)**: 도착과 충돌을 대칭으로 설정해 두 결과의 무게를 같게 만든다.

2. **시간 페널티 = 빙빙 돌기 억제 (-0.01)**: 종료 보상보다 훨씬 작아야 "느리게라도 도착하는 것"이 여전히 이익이 되도록 유지한다. 너무 크면 드론이 빠른 실패를 선택한다.

3. **거리 보상 = 즉각 방향 신호 (±0.01)**: 시간 페널티와 같은 크기로 맞춰서, 잘 가면 패널티를 상쇄(합계 0), 잘못 가면 두 배 패널티(합계 -0.02)가 되게 한다. 종료 보상보다 100배 작아야 드론이 "경로 최적화"가 아닌 "도착"을 최종 목표로 삼는다.

# 04 FGSM vs PGD 차이

# FGSM vs PGD 공격 코드 비교 설명

> **핵심 개념을 한 줄로:**
> FGSM은 "틀린 방향을 한 번에 크게 밀기"고,
> PGD는 "틀린 방향을 조금씩 여러 번 밀되, 매번 제자리를 확인하고 조이기"다.

---

## 두 함수를 나란히 놓고 비교

### FGSM (62~78번째 줄) — 1번짜리

```python
def fgsm_attack(model, img_tensor, targets, epsilon):
    if epsilon == 0:
        return img_tensor.clone()           # 공격 없음

    adv = img_tensor.clone().detach().requires_grad_(True)
    model.train()

    # ① 딱 1번: 손실 계산
    total_loss = sum(model([adv], targets).values())
    model.zero_grad()

    # ② 딱 1번: 기울기 계산
    total_loss.backward()

    # ③ 딱 1번: 이미지 업데이트 (epsilon 만큼 한 방에)
    return torch.clamp(
        img_tensor + epsilon * adv.grad.data.sign(),
        0.0, 1.0
    ).detach()
```

---

### PGD (81~118번째 줄) — N번짜리

```python
def pgd_attack(model, img_tensor, targets, epsilon, steps=PGD_STEPS):
    if epsilon == 0:
        return img_tensor.clone()           # 공격 없음

    step_size = epsilon / 4                 # 한 번에 이동할 최대 거리
    x_orig = img_tensor.detach()            # 원본 이미지 저장
    x_adv  = img_tensor.clone().detach()    # 공격 이미지 (변해감)

    model.train()
    for _ in range(steps):                  # ← 이 for 루프가 핵심 차이!

        x_input = x_adv.detach().requires_grad_(True)

        # ① N번: 손실 계산
        total_loss = sum(model([x_input], targets).values())
        model.zero_grad()

        # ② N번: 기울기 계산
        total_loss.backward()

        with torch.no_grad():
            # ③ N번: 이미지 조금씩 업데이트 (step_size만큼만)
            x_adv = x_adv + step_size * x_input.grad.sign()

            # ④ N번: epsilon 범위 초과하면 도로 잘라냄 (투영)
            x_adv = torch.max(
                torch.min(x_adv, x_orig + epsilon),
                x_orig - epsilon
            )

            # ⑤ N번: 픽셀 유효 범위 [0,1] 유지
            x_adv = torch.clamp(x_adv, 0.0, 1.0)

    return x_adv.detach()
```

---

## "한 번"과 "여러 번"의 차이를 만드는 코드 줄

정확히 이 두 부분이 전부다.

```python
# ─────────── FGSM ───────────
# for 루프 없음 — 그냥 한 줄
return torch.clamp(img_tensor + epsilon * adv.grad.data.sign(), 0.0, 1.0)
#                  ↑ epsilon 전체를 한 방에 더함

# ─────────── PGD ────────────
for _ in range(steps):              # ← 이 줄만 감싸면 PGD
    ...
    x_adv = x_adv + step_size * x_input.grad.sign()
    #               ↑ step_size(= epsilon/4) 씩만 더함, 여러 번
    x_adv = torch.max(torch.min(x_adv, x_orig + epsilon), x_orig - epsilon)
    #        ↑ 투영: 원본에서 epsilon 이상 멀어지면 잘라냄
```

FGSM을 PGD로 바꾸는 데 필요한 변경은 사실 세 가지뿐이다:
1. `epsilon` 대신 `step_size = epsilon / 4` 사용
2. `for _ in range(steps):` 루프로 감싸기
3. 루프 안에 L∞ 투영 한 줄 추가

---

## FGSM vs PGD 동작 방식을 그림으로

```
공격의 목표: "원본(●)에서 최대 epsilon 이내에서
              AI가 가장 틀리는 지점(★)을 찾는 것"

epsilon 범위 (원통형 공간):
  ┌───────────────────────┐
  │          ★            │  ← 가장 틀리는 지점 (우리가 원하는 곳)
  │                       │
  │       ←←←←←          │
  │                       │
  │●                      │  ← 원본 이미지
  └───────────────────────┘

FGSM: 기울기 방향으로 한 번에 epsilon만큼 이동
  ●───────────────────────►
  시작점              도착점 (epsilon 경계)
  문제: 기울기는 현재 위치에서만 계산됨
        → 도착점이 ★과 다른 방향일 수 있음

PGD: epsilon/4씩 10번 이동하며 매번 기울기 재계산
  ●→→→→→→★
  각 → 이후 기울기를 다시 계산해 방향 수정
  → ★에 더 가깝게 도달 가능
```

---

## step_size = epsilon / 4 로 정한 이유

**코드 (100번째 줄):**
```python
step_size = epsilon / 4
```

이 값이 왜 epsilon의 1/4인지, 세 가지 측면에서 설명한다.

### ① 너무 크면 (step_size = epsilon): FGSM과 같아진다

```
step_size = epsilon 이면:

  1번째 걸음: x_adv = x_orig + epsilon * sign(grad)
  투영:       clip(x_adv, x_orig ± epsilon) → 이미 경계에 붙음
  2~10번째:  이미 경계에 있어서 방향만 바뀔 뿐 의미 없음

→ 결국 FGSM과 동일한 결과
```

### ② 너무 작으면 (step_size = epsilon / 100): 경계까지 못 간다

```
10번 × (epsilon/100) = epsilon/10

→ 10번 이동해도 epsilon 경계의 10%밖에 탐색 못 함
→ 공격 강도가 약해짐
```

### ③ epsilon / 4 의 균형점

```
10번 × (epsilon/4) = 2.5 × epsilon

경계(epsilon) 안에서 2.5배 거리를 탐색할 수 있다.
투영으로 경계 밖은 잘리지만, 경계 안을 충분히 탐색.

    ←─────── epsilon ────────→
    ←──────────────── 2.5×epsilon 탐색 시도 ──────────────────→
    (경계 밖은 잘림)     (경계 안을 여러 각도에서 탐색 가능)
```

epsilon/4는 "충분히 많은 방향을 탐색하면서도, 경계 안에서 의미 있는 이동"을 가능하게 하는 실용적 기준값이다.

## steps = 10 으로 정한 이유

**코드 (44번째 줄):**
```python
PGD_STEPS = 10
```

### 실험 결과로 확인한 수렴 패턴

PGD에서 반복 횟수와 공격 강도의 관계는 대략 이렇다:

```
반복 횟수   공격 강도(개념적)   비고
─────────────────────────────────────
1번         (= FGSM 수준)      단순 방향 한 번
3번         꽤 강해짐          주요 방향 탐색 완료
5번         거의 수렴          추가 이득 작아짐
10번        충분한 수렴        ← 우리가 사용
20번 이상   거의 차이 없음     계산 비용만 증가
```

### 계산 비용과의 관계

FGSM은 이미지 1장당 계산을 **1번** 한다.
PGD_STEPS=10이면 이미지 1장당 계산을 **10번** 한다.
즉, PGD는 FGSM보다 **10배 느리다.**

10번으로 설정한 이유: "충분한 공격 강도를 얻으면서 계산 비용을 합리적으로 유지"하는 표준 설정.
학술 논문에서도 PGD 기본 설정으로 steps=10~40이 자주 쓰인다.

## 실험 결과: PGD가 FGSM보다 2.4~3.6배 강한 이유

`attack_results.json`과 `attack_results_pgd.json`에서 측정된 평균 bbox 오차:

| epsilon | FGSM 평균 오차 | PGD 평균 오차 | PGD / FGSM 비율 |
|---------|--------------|-------------|----------------|
| 0.01 | 12.95 px | 30.97 px | **2.4배** |
| 0.03 | 13.56 px | 36.88 px | **2.7배** |
| 0.05 | 14.28 px | 36.03 px | **2.5배** |
| 0.07 | 12.29 px | 42.62 px | **3.5배** |
| 0.10 | 10.54 px | 38.30 px | **3.6배** |

**왜 epsilon이 커질수록 배율이 올라가는가?**

```
epsilon이 작을 때 (0.01):
  epsilon-ball이 좁음 → 탐색 공간이 작음
  FGSM도 한 번에 epsilon 경계에 도달 가능
  → FGSM과 PGD의 차이가 작음 (2.4배)

epsilon이 클 때 (0.07~0.1):
  epsilon-ball이 넓음 → 탐색 공간이 큼
  FGSM은 한 방향으로만 이동 → 최악의 지점(★)을 못 찾음
  PGD는 여러 번 방향 수정 → 넓은 공간에서 ★을 찾아냄
  → 차이가 커짐 (3.5~3.6배)
```

수치로 보면:

```
epsilon=0.01일 때:
  탐색 공간: ±0.01 (픽셀 강도 ±1/100)
  FGSM: 한 번에 ±0.01 이동 → 경계 도달
  PGD: 0.01/4 = 0.0025씩 10번 → 경계 안 탐색 (공간이 좁아 차이 작음)

epsilon=0.07일 때:
  탐색 공간: ±0.07 (픽셀 강도 ±7/100)
  FGSM: 한 번에 ±0.07 이동 → 기울기 방향 단순 도달
  PGD: 0.07/4 = 0.0175씩 10번 → 공간이 넓어 방향 수정 효과 극대화
```

## 두 공격이 DQN 드론 실험에 미친 영향

우리 실험에서는 DQN 드론 학습에 **PGD (ε=0.03)** 를 사용했다.
이유: FGSM(ε=0.03)의 평균 오차 13.56px보다 PGD의 36.88px이
드론 항법에 더 현실적인 위협 수준이기 때문이다.

```
FGSM ε=0.03: 평균 13.56px → 임계값 10px 기준으로 대부분 CE 판정
PGD  ε=0.03: 평균 36.88px → 이진 CE(>10px) 기준으로 거의 전량 폐기
                           → 연속 CE에서도 저장 확률 max(0.05, 1-36.88/50) = 26%

PGD를 사용했기 때문에:
  V4-A 이진 CE: 82.8% 폐기 (shift 대부분 10px 초과)
  V4-B 연속 CE: 27.9% 폐기 (shift 비례 저장으로 일부 살림)
```

만약 FGSM으로 실험했다면 평균 shift가 더 작아 CE 폐기율이 낮았을 것이고,
V4-A/B의 차이도 덜 두드러졌을 것이다.

## 최종 요약

| 항목 | FGSM | PGD |
|------|------|-----|
| 반복 횟수 | 1번 | 10번 (`PGD_STEPS = 10`) |
| 한 번 이동 크기 | epsilon 전체 | epsilon / 4 (`step_size`) |
| 투영(범위 제한) | 없음 | 매 스텝 후 epsilon-ball로 잘라냄 |
| 계산 비용 | 빠름 (1x) | 느림 (10x) |
| 공격 강도 | 기준 | **2.4~3.6배 강함** |
| 핵심 코드 차이 | 루프 없음 | `for _ in range(steps):` 한 줄 |

**step_size = epsilon/4의 의미**: 한 번에 경계까지 안 가고, 조금씩 여러 번 가면서 매번 "지금 어느 방향으로 가야 가장 틀리게 만드는가"를 다시 계산한다.

**steps = 10의 의미**: 3~5번 이후에는 수렴하지만, 10번이 "충분한 탐색 + 합리적 계산 비용"의 표준 균형점이다.

## 원인 1: 항상 같은 길만 연습 → 과적합(Overfitting)

V3의 드론은 1,000번 내내 딱 **하나의 경로**만 연습했다.

```
목적지 (13.5, 16.5) ← 우상단 구석
         ↑
    [장애물 벽-2]  ← 항상 이 벽을 통과해야 함
         ↑
    [장애물 벽-1]  ← 항상 이 벽도 통과해야 함
         ↑
시작점 (1.0, 1.0) ← 좌하단 구석
```

1,000번 모두 이 좌하단→우상단 경로다.
드론은 "이 특정 경로에서 어떻게 움직여야 하는가"만 학습한다.

문제는 이 경로가 **두 개의 장애물 벽 사이 좁은 틈**을 반드시 통과해야 한다는 것이다.

```
      ●   ●   ●    ← wall-2 (3개 장애물)
   ↑
이 좁은 틈(약 0.5km)을 반드시 통과해야 함
   ↑
      ●   ●   ●    ← wall-1 (3개 장애물)
```

---

## 원인 2: 과적합 후 망각(Catastrophic Forgetting)

DQN 학습의 초기에는 탐색(exploration)이 많다.
엡실론 값이 높아서 드론이 이곳저곳 무작위로 날아다닌다.
그러다 우연히 좁은 틈을 통과해 목적지에 도달하면 → 그 경험을 저장한다.

에피소드 400쯤 되면 "이 특정 경로에서의 통과법"을 어느 정도 익힌다 → 47%.

하지만 에피소드가 더 쌓이면 엡실론이 낮아지고, 탐색보다 **활용(exploitation)** 이 많아진다.
이때 리플레이 메모리가 꽉 차서 오래된 경험이 밀려나기 시작한다.

```
메모리 한도: 10,000개

에피소드 400 이전:
  [경험1: 틈 통과 성공] [경험2: ...] ... [경험10000: ...]
  ← 성공 경험이 메모리에 살아있음

에피소드 600 이후:
  새 경험들이 쏟아지면서 성공 경험들이 하나씩 밀려남
  [경험 1001: 실패] [경험 1002: 실패] ... [경험 10000: 실패]
  ← 성공 경험이 모두 사라짐 → 도착률 붕괴
```

같은 경로만 반복하기 때문에, 새로 들어오는 경험도 "비슷한 실패 경험"이다.
결국 메모리가 실패 경험으로 채워지고 도착률이 0%가 된다.

```python
class DroneEnvN:

        self,

        fallback_pos: list[float] | None = None,  # 랜덤 실패시 비상 좌표
```

**reset() 함수 (107~112번째 줄) — V3와 핵심적으로 다른 부분:**

```python
def reset(self) -> list[float]:

    self.heading = random.uniform(0.0, 360.0)
```

V3의 `list(START_V3)` / `list(GOAL_V3)` 가 있던 자리에,
V4에서는 `_random_free_pos()` 호출이 들어간다.

**랜덤 위치 생성 함수 (117~132번째 줄):**

```python
def _random_free_pos(
    self,
    min_dist_from: list[float] | None = None,
    min_dist: float = 0.0,
    max_tries: int = 2000,            # 최대 2000번 시도
) -> list[float]:
    for _ in range(max_tries):
        x = random.uniform(0.5, SPACE_W - 0.5)   # ← 0.5~14.5 사이 랜덤 x
        y = random.uniform(0.5, SPACE_H - 0.5)   # ← 0.5~17.5 사이 랜덤 y
        if self._min_obs_dist(x, y) <= OBS_RADIUS * 2:
            continue   # 장애물과 너무 가까우면 다시 뽑기
        if min_dist_from is not None:
            if math.dist([x, y], min_dist_from) < min_dist:
                continue   # 시작점과 너무 가까우면 다시 뽑기
        return [x, y]      # 조건 만족하면 이 좌표 사용
    return list(self.fallback_pos)  # 2000번 실패하면 (1.0,1.0) 반환
```

조건:
1. 공간 경계 안쪽 (0.5~14.5, 0.5~17.5)
2. 모든 장애물과 3.0km(= OBS_RADIUS×2) 이상 떨어져야 함
3. 시작점과 목적지는 서로 3.0km 이상 떨어져야 함 (`min_dist=3.0`)

## 왜 랜덤 경로가 더 잘 학습되는가

### 고정 경로의 문제: 한 가지 상황만 배움

```
V3에서 드론이 배운 것:
  "(1.0, 1.0)에서 출발해 (13.5, 16.5)로 가는 방법"

테스트할 때:
  같은 (1.0, 1.0) → (13.5, 16.5) 경로이므로 처음엔 성공

학습이 깊어질수록:
  메모리가 이 경로의 실패 경험으로 채워짐
  → "이 특정 경로에서 어떻게 피해야 하는가"조차 잊어버림
```

### 랜덤 경로의 장점: 일반적인 항법 능력 배움

```
V4/env_test에서 드론이 배운 것:
  에피소드 1:   (2.3, 3.1) → (11.7, 14.2) 경로
  에피소드 2:   (8.1, 1.5) → (3.2, 16.8) 경로
  에피소드 3:   (0.8, 8.3) → (13.1, 5.6) 경로
  ...
  에피소드 1000: 또 다른 새 경로

각 경로마다 다른 각도에서 장애물 벽을 만남
→ 드론이 "장애물 벽을 어떻게 피하는가"라는 일반 원리를 배움
→ 새로운 경로에서도 그 원리를 적용 가능
```

이것을 **일반화(generalization)** 라고 한다.
시험 문제 유형을 외우는 것이 아니라, 수학 원리 자체를 이해하는 것과 같다.

## env_test.py의 실험 설계: V3 실패 원인 분리

`env_test.py`는 사실 "v3 실패의 원인이 장애물 수 때문인가, 고정 경로 때문인가"를 알아내기 위해 만든 파일이다. 코드 맨 위의 설명에 이렇게 나와 있다:

```python
"""환경 단독 테스트: 5장애물/랜덤 vs 8장애물/랜덤.

실험 목적:
  v3 실험(8장애물 + 고정 시작/목적지)의 성능 저하 원인 분리.
  - 가설 A: 장애물 수 자체(8개)가 문제
  - 가설 B: 고정 경로(항상 양쪽 벽 통과)가 문제
"""
```

두 가지를 나란히 비교했다:

| 실험 이름 | 장애물 수 | 경로 방식 | 최종 도착률 |
|-----------|-----------|-----------|-------------|
| env_test 환경-1 | **5개** | 랜덤 | ~72% |
| env_test 환경-2 | **8개** | 랜덤 | **74%** |

결과: **8개 장애물 + 랜덤 경로로도 74% 달성**

이것이 결정적인 증거다.
장애물 수(8개)는 문제가 아니었다.
**고정 경로**가 문제였다.

```
V3 실험 (1000 에피소드):
  장애물: 8개  시작: (1.0,1.0) 고정  목적지: (13.5,16.5) 고정
  결과:   ep≈400에서 47% → 최종 0%

      원인 분석
         ↓
  env_test 실험 (1000 에피소드):
  장애물: 8개  시작: 랜덤  목적지: 랜덤
  결과:   최종 74%

      "장애물 수가 아니라 고정 경로가 문제" 확인
         ↓

  장애물: 8개 (V3와 동일)  시작: 랜덤  목적지: 랜덤  + CE 필터
  결과:   V4-A (이진 CE) 31%  / V4-B (연속 CE) 65%
         ↑
         고정→랜덤 전환으로 V3 0%에서 복구

  V5 실험 (1000 에피소드):
  장애물: 8개  시작: 랜덤  목적지: 랜덤  + CE 필터 + 시간/거리 보상
  결과:   V5-A (이진 CE) 49%  / V5-B (연속 CE) 78%
```

## 최종 요약: V3 실패의 진짜 원인

| 의심 원인 | 실제 영향 | 근거 |
|-----------|-----------|------|
| 장애물이 8개로 늘었다 | **아님** | 8개+랜덤으로 74% 달성 (env_test) |
| **고정 경로를 썼다** | **맞음** | 고정에서 0%, 랜덤으로 바꾸자 74% |
| CE 필터가 없었다 | 부분적 | CE는 V4에서 추가 (별도 효과) |

**V3 실패 = 고정 경로 과적합 + 메모리 교체로 인한 성공 경험 소멸**

코드로 바꾼 것은 두 줄이었지만, 결과는 0% → 65%+(V4-B) 의 차이를 만들었다.
