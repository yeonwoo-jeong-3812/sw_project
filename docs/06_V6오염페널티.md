# V6: 오염 심각도 페널티를 학습 루프에 연결

> **핵심 개념을 한 줄로:**
> V5-B가 "오염된 경험은 저장하지 말자"는 방어였다면,
> V6는 "오염된 상황에서 행동할 때 추가로 점수를 깎자"는 적극적 억제다.

---

## V6가 하려는 것

V5-B까지는 PGD 공격으로 shift가 크면 그 경험을 **저장하지 않는** 방식이었다.
그런데 "저장 안 하기"만으로는 드론이 오염된 상황에서 어떻게 행동해야 하는지 배울 수 없다.

V6는 한 가지를 더 추가한다:
- 오염이 심각한 상황(RISK/CRITICAL)에서 행동하면 **보상에서 추가 페널티를 깎는다**
- 드론이 "카메라 신호가 이상할 때는 장애물에 가까이 가지 말자"를 직접 학습하게 한다

---

## 오염 심각도 3단계

```
shift(px) 기준:

  0px ─────── 10px ──────── 20px ──────────→
       STABLE       RISK          CRITICAL

  STABLE   (shift ≤ 10px): 카메라 신호 정상 → 추가 페널티 없음
  RISK     (10 < shift ≤ 20px): 경미한 오염 → 조건부 페널티 -0.01
  CRITICAL (shift > 20px):     심각한 오염 → 무조건 페널티 -0.03
```

**실제 코드:**

```python
# dqn_comparison_v6.py
SHIFT_THRESH_RISK     = 10.0   # px
SHIFT_THRESH_CRITICAL = 20.0   # px

def _assess_shift(shift: float) -> CorruptionLevel:
    if shift <= SHIFT_THRESH_RISK:
        return CorruptionLevel.STABLE
    elif shift <= SHIFT_THRESH_CRITICAL:
        return CorruptionLevel.RISK
    else:
        return CorruptionLevel.CRITICAL
```

우리 실험(PGD ε=0.03)에서 shift 분포:
- STABLE(≤10px): 약 17%
- RISK(10~20px): 약 77%
- CRITICAL(>20px): 약 6%

---

## 페널티 크기 선택 과정

### 1차 시도: 너무 컸다

처음에 RISK=-0.02로 시험 실행했더니 ep200 도착률이 1%로 붕괴했다.

이유를 분석해보니:
- RISK shift 구간이 전체 스텝의 77% → RISK 발동률이 36.6%/스텝
- 매 비종료 스텝 평균 페널티: 0.366 × (-0.02) = **-0.0073/스텝**
- 거기에 시간 페널티(-0.01) + 거리 페널티(-0.01) 합산 시 **-0.028/스텝**
- 목표 도착(+1.0)이 너무 멀게 느껴져 드론이 포기 학습

### 2차 조정: 거리 조건 추가 + 크기 축소

두 가지를 동시에 바꿨다:

| 항목 | 1차 | 2차 (최종) |
|------|-----|-----------|
| RISK 페널티 크기 | -0.02 | **-0.01** |
| CRITICAL 페널티 크기 | -0.05 | **-0.03** |
| RISK 발동 조건 | shift만 | shift + 장애물 접근 중 + **obs_dist < 3km** |

**조건이 추가된 코드:**

```python
# dqn_comparison_v6.py — 비종료 스텝 보상 블록
if corruption is CorruptionLevel.RISK:
    # RISK 페널티: 3가지 조건 모두 만족해야 발동
    if (prev_obs_dist is not None           # 이전 거리 기록 있음 (첫 스텝 제외)
            and obs_dist < prev_obs_dist    # 장애물에 가까워지는 중
            and obs_dist < OBS_DIST_RISK_THRESH):  # 3km 이내
        reward += PENALTY_RISK   # -0.01
        risk_count += 1

elif corruption is CorruptionLevel.CRITICAL:
    # CRITICAL 페널티: 조건 없이 무조건 발동
    reward += PENALTY_CRITICAL   # -0.03
    critical_count += 1
```

왜 3km 조건을 추가했는가?
- "장애물이 멀리 있을 때 카메라가 흔들려도 드론에게 별 위험이 없다"
- 가까울 때만 억제해서 페널티 발동 빈도를 줄이고 학습 안정성을 확보

## V6 전체 보상 구조

```
매 걸음마다:

┌─────────────────────────────────────────────────────────┐
│ 종료 상황 (V5와 동일)                                    │
│   목적지 도달 → +1.0                                     │
│   충돌·이탈   → -1.0                                     │
├─────────────────────────────────────────────────────────┤
│ 비종료 상황                                              │
│   시간 페널티          → -0.01 (항상)                   │
│   거리 보상            → +0.01 (목적지 접근 시)          │
│   거리 패널티          → -0.01 (목적지 후퇴 시)          │
│   [V6 추가] RISK 페널티   → -0.01 (조건부: 3km+접근+RISK) │
│   [V6 추가] CRITICAL 페널티 → -0.03 (무조건: CRITICAL)  │
└─────────────────────────────────────────────────────────┘
```

CE 경험 저장 방식은 V5-B와 완전히 동일하다:
```python
ratio = max(0.05, 1.0 - shift / 50.0)   # 연속 가중치 (V5-B와 같음)
if random.random() < ratio:
    agent.push(c_state, action, reward, c_next, done)
```

## V5-B 거리 보상 실제 측정

V6를 설계하기 전, "V5-B 학습된 정책이 실제로 얼마나 목표를 잘 향해 가는가"를 직접 측정했다.

측정 방법: v5b_policy.pth 가중치 로드 → epsilon=0.05 → 200 에피소드 추론 (학습 없음)

결과:
- 비종료 스텝 중 목표 접근: **96.6%**
- 비종료 스텝 중 목표 후퇴: **3.4%**
- 평균 (시간+거리)/스텝: **-0.00079** (거의 0에 가까움)

이 측정이 중요한 이유:
- 처음 설계할 때 "거리 보상이 평균 -0.02/스텝"이라고 가정했지만, 이는 **30배 과대 추정**이었다
- 학습이 잘 된 V5-B 정책은 대부분 목표 방향으로 이동하므로 거리 보상이 거의 상쇄됨
- V6 페널티 크기를 결정할 때 이 실측값을 기준으로 삼았다

## 전체 학습 과정에서 추적한 지표

V6는 매 100에피소드마다 다음을 출력한다:

```
Ep  100 | 도착률=0.000 | 충돌률=1.000 | ε=0.606 
       | CE폐기=1254(75.1%) | 유효경험=416
       | RISK=300(17.9%) CRIT=102(6.1%)
```

| 지표 | 설명 |
|------|------|
| CE폐기 | shift에 비례해 확률적으로 버린 경험 수 |
| 유효경험 | 실제 리플레이 메모리에 저장된 경험 수 |
| RISK=300(17.9%) | RISK 페널티가 전체 스텝의 17.9%에서 발동 |
| CRIT=102(6.1%) | CRITICAL 페널티가 6.1%에서 발동 |

## 본 실행 결과 (1000 에피소드)

| 항목 | V5-B | V6 |
|------|------|-----|
| 최종 도착률 | 78.0% | **79.0%** |
| 최종 충돌률 | 22.0% | 21.0% |
| 도착 에피소드 수 | 574개 | **574개** |
| RISK 발동률 | - | 26.5%/스텝 |
| CRITICAL 발동률 | - | 5.9%/스텝 |

**도착 에피소드 수가 V5-B와 정확히 같다(574개)**는 점이 주목할 만하다.
- V6 보상 구조가 V5-B에 비해 불리한 면도(페널티 추가), 유리한 면도(더 나은 회피 학습) 있어 결과가 비슷하다
- 최종 도착률은 +1.0%p 향상됐지만, 큰 개선이라고 보기 어렵다

## 왜 V6의 개선이 크지 않은가

페널티 자체는 잘 작동했다. RISK 발동 조건(3가지 동시 만족)이 실제로 26.5%/스텝에서 발동했고, 이는 적절한 범위다.

그런데 왜 성능 개선이 +1.0%p에 그쳤는가?

```
V6의 한계:
  리플레이 메모리 = 기존 ReplayMemory (DQNAgent.memory)
                 = 모든 경험을 구분 없이 하나의 큐에 저장

  즉, CE 경험 저장 방식은 V5-B와 완전히 동일
  → 페널티 신호만 추가됐을 뿐, 경험 품질 관리는 미개선

  다음 과제 = ImprovedDDMBuffer로 경험을 타입별로 분리
           = RE(결과)/DE(위험)/SE(안전)/CE(오염) 4분류
           → V7에서 구현
```

---

## V6의 위치

```
V5-B (연속 CE 가중치 저장)
     │
     ├─ CE 경험을 shift에 비례해 확률적 폐기  ← 이미 있음
     │
     + [V6 추가] 오염 심각도에 따른 보상 페널티
       RISK:     접근중 + 3km 이내 → -0.01
       CRITICAL: 무조건           → -0.03
     │
     결과: 78% → 79% (+1.0%p)
     │
     V7에서 할 일:
       리플레이 버퍼를 타입별로 분리 (ImprovedDDMBuffer)
       RE:DE:SE=4:3:3 비율로 샘플링
       CE는 버퍼에 아예 저장하지 않음
```
